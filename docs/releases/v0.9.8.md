# MCPlexer v0.9.8

MCPlexer v0.9.8 is a public `AGPL-3.0-or-later` release. It unifies the Code
Mode concurrent calling convention with the sequential one, sharpens
operational monitoring, and improves memory-recall ranking.

## Highlights

- **Code Mode — one calling convention for one and many calls.** `parallel()`
  now accepts bound tool references in the same form you already call tools
  sequentially, alongside the existing string descriptor:

  ```js
  parallel([[github.list_issues, {repo}], [linear.search, {query}]]);  // [fn, args] tuple
  parallel([{tool: github.list_issues, args: {repo}}, ...]);           // object form, bound ref
  parallel([{tool: "github__list_issues", args: {repo}}, ...]);        // legacy string (unchanged)
  ```

  Each tool function carries its canonical name in a hidden, non-enumerable
  property, so the concurrent path recovers the name and dispatches with the
  same real goroutine concurrency as before — only the surface changes. No-arg
  concurrent calls now send `{}` instead of `null`, matching the sequential
  path so both forms send identical shapes for identical calls. The change is
  backward compatible; `help()` and the `execute_code` tool descriptions
  advertise the new form.

- **Operational awareness for monitoring.** Baseline learning, incident
  distillation, and a usage summary give the monitoring surface a clearer view
  of what "normal" looks like and a more compact account of what happened.

- **Memory recall ranking.** Recall bounds recency so it complements relevance
  instead of dominating it, and the ranking evaluation harness was rebuilt to
  model production conditions.

## Upgrade notes

- Back up the MCPlexer application state before upgrading. This release adds no
  new database migrations over v0.9.7.
- The daemon upgrade is an in-place atomic swap with drain-wait, readiness
  verification, and automatic rollback (`task upgrade`). Expect roughly one to
  two seconds of downtime.
- The Code Mode change is additive and backward compatible: existing
  `parallel([{tool: "ns__name", args}])` scripts continue to work unchanged.
